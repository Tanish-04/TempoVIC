# TempoVIC Training Pipeline

This directory contains the complete training pipeline for **TempoVIC**, encompassing data pre-processing, graph assembly, model definitions (including ablation variants), and the two-phase training process.

---

## Terminology & Pipeline Overview

As described in our paper, the TempoVIC approach tackles Vulnerability-Inducing Commit (VIC) identification in two sequential stages, each building on the output of the previous.

**Stage 1 — Root-Cause Line Identification:** Given a patched file, the model first learns to rank and surface the *deletion lines* most likely to represent the root cause of the vulnerability. Identifying these lines narrows the search space and provides a fine-grained, code-level signal.

**Stage 2 — VIC Ranking:** Using the root-cause line embeddings produced in Stage 1 as anchors, the model then ranks historical commits to surface the one most likely to have *introduced* the vulnerability (i.e., the VIC).

To keep the codebase and documentation concise, we adopt the following shorthand throughout:

| Paper Term | Codebase Term | What it does |
|---|---|---|
| Root-Cause Line Identification | **Phase 1** | Trains on deletion-line pairs to learn which lines caused the vulnerability |
| VIC Ranking | **Phase 2** | Uses Phase 1 embeddings to rank commits and identify the VIC |

Wherever you see *Phase 1* or *Phase 2* in configs, scripts, or logs, refer back to this table to map them to their conceptual roles.

## 2. Directory Structure

The codebase is organized hierarchically to separate data logic, model architectures, and training loops.

```text
training_pipeline/
├── build_graphs.py          # Unified script to pre-compute temporal and non-temporal graphs from raw data.
├── config_utils.py          # Helper for loading and parsing the training_config.yaml file.
├── main.py                  # Main entry point for training (runs Phase 1 & Phase 2).
├── training_config.yaml     # Central configuration for hyper-parameters, data paths, and model selection.
├── data_processing/         # Data loading, graph construction, and batching logic.
│   ├── dataset.py           # PyTorch datasets (DeletionLineDataset & CommitRankingDataset).
│   └── phase1/
│       ├── minigraph.py     # BFS/DFS logic to extract localized CFG/DFG neighbourhoods.
│       ├── pairs.py         # Logic for constructing contrastive deletion-line pairs.
│       └── processing.py    # Core graph logic (building CFG/DFG edges, PyG conversion).
├── models/                  # Neural network architecture definitions.
│   ├── base.py              # Base classes (BaseEncoder, BasePhase1Model) to prevent code duplication.
│   ├── phase1_model.py      # Phase 1 model: Root-Cause Line Ranking.
│   ├── phase2_model.py      # Phase 2 model: VIC Ranking via dual query attention.
│   ├── shared_encoder.py    # GAT-based encoder shared between both phases with UnixCoder base.
│   └── variants/            # Ablation study architectures.
│       ├── deepsets/        # Order-agnostic DeepSets variant (no temporal edges).
│       └── transformer/     # Section-wise Transformer variant (no temporal edges).
└── training/                # Training loops, loss functions, and evaluation logic.
    ├── embedding_cache.py   # Caches Phase 1 embeddings for efficient Phase 2 commit ranking.
    ├── evaluation.py        # Global Top-K VIC ranking metrics and evaluation framework.
    ├── loss.py              # Custom loss functions (PairwiseRankingLoss, LabelSmoothingRankingLoss).
    ├── phase1_trainer.py    # Training loop and optimization for Phase 1 (Root-Cause Line Identification).
    ├── phase2_trainer.py    # Training loop and optimization for Phase 2 (VIC Ranking).
    └── utils.py             # General utilities (seed setup, early stopping, model builders).
```

## 3. Configuration

All primary settings (paths, hyper-parameters, and model choices) are centralized in **`training_config.yaml`**.

Key parameters you may want to change:

| Parameter | Description |
|---|---|
| `paths.data_root` | Path to the raw JSON graphs generated by graph construction |
| `model.encoder_type` | Controls which architecture to train: `gat`, `deepsets`, or `section_transformer` |
| `paths.graph_mode` | Should match the pre-built graph dataset to load: `full_graph` or `no_temporal` |

## 4. Pipeline Execution

The pipeline is split into two independent steps: **Graph Pre-building** and **Training**.

### Step 1 — Pre-computing Graph Structures

Before training, raw graphs must be extracted and assembled into PyG-compatible subgraphs. A single script handles this for both the main architecture and ablation variants.

```bash
# Build the temporal graph structure for the main GAT model
python build_graphs.py --mode full_graph

# Build the non-temporal graph structure for ablation variants (DeepSets / Transformer)
python build_graphs.py --mode no_temporal

# Build both at once
python build_graphs.py --mode all
```

This populates `temporal_graph/full_graph/` and/or `temporal_graph/no_temporal/` under your configured `output_dir`.

### Step 2 — Training

Once the graphs are built, execute the full training pipeline. `main.py` automatically runs **Phase 1** (Root-Cause Line Identification), evaluates it, caches the resulting embeddings, and then runs **Phase 2** (VIC Ranking) on top of them.

> **Before running:** ensure `encoder_type` and `graph_mode` in `training_config.yaml` match the architecture you intend to train (see the table in [Section 5](#5-model-variants)).

```bash
# Run the complete two-phase training loop
python main.py

# Override specific hyper-parameters via CLI
python main.py --encoder-type deepsets --graph-mode no_temporal --phase1-epochs 30 --phase2-lr 0.0005
```

To run only Phase 2 (VIC Ranking) using a pre-existing Phase 1 checkpoint:

```bash
python main.py --skip-phase1 --phase1-checkpoint-dir checkpoints_XYZ
```

## 5. Model Variants

Three architectural variants are supported, selectable via `model.encoder_type` in the config:

| `encoder_type` | Role | Graph Mode | Description |
|---|---|---|---|
| `gat` | **Main Model** | `full_graph` | `SharedEncoder` with Graph Attention layers over temporal + structural edges (CFG/DFG + temporal). |
| `deepsets` | Ablation | `no_temporal` | Order-agnostic pool-and-refine architecture; memory-efficient, no temporal edges. |
| `section_transformer` | Ablation | `no_temporal` | Standard sequential Transformer encoder operating over commit sections; no temporal edges. |

The ablation variants (`deepsets`, `section_transformer`) isolate the contribution of temporal edges by operating on `no_temporal` graphs, making them directly comparable to the main `gat` model.
